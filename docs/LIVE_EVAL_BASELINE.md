# v0.3 真实盲测验收

本记录对应可证明硬约束执行闭环 v0.3 的冻结合同盲测。六个案例保留原有前提、人物和 rubric，但为每例冻结了可审计的 ChapterOutline 与 ChapterContract；不再让供应商在每次运行时随机改写被测约束。

原始真实供应商证据位于 `evals/live_results/v03x_frozen_full/`，每例保留基线、受约束生成、两种展示顺序的盲审结果与调用元数据。`summary.recomputed.json` 由当前聚合协议从这些原始文件重算，不重新调用供应商或改写原始试验。

## 验收结果

| 门槛 | 目标 | 结果 | 状态 |
| --- | ---: | ---: | --- |
| 全部硬约束通过率 | ≥90% | 100%（6/6） | 通过 |
| 交换顺序一致率 | ≥80% | 83.3%（5/6） | 通过 |
| 平均额外生成调用 | ≤5 | +4.5 | 通过 |
| 平均额外生成 token | ≤15,000 | +11,532 | 通过 |
| 本地回归 | 全部通过 | 71 pytest + 36 evals | 通过 |

生成路径平均调用为 5.5 次、12,948 token；直接基线为 1 次、1,416 token。评测所需的两次盲审不计入产品生成成本。

## 盲审协议

- 每个候选以 `baseline,treatment` 和 `treatment,baseline` 两种顺序各评一次，原始结果完整保留。
- 两次都确认的硬违规才计入外部硬失败；只在一侧出现的违规作为 `judge_disputed_hard_failures` 保留，不能推翻另一侧的明确审查结果。
- 两次均平局，或一侧平局、另一侧为同方向小胜，属于非反向的一致结果；一侧基线胜而另一侧流程胜才是顺序反转。
- 可用 `python -m evals.live_quality --recompute-dir evals/live_results/v03x_frozen_full` 在不消耗供应商额度的情况下重算当前协议摘要。

质量胜率结论仍是 `no_clear_improvement`：6 例中 5 例为平局、1 例存在顺序反转。因此本次验收证明的是硬约束执行与成本门槛，而不是“流程文笔必然优于直接写作”。
